Minidecaf Tutorial

💡 Credit to @[Saltyp0rridge](https://bowenbryanwang.github.io/), fixed by Chenytang Zhao 😵‍💫

本文档对 2022 秋编译原理课程实验框架进行浅浅的解读，并对 stage 1-3 进行简要指引。在理解实验框架的基础上完成实验是十分简单的！

MiniDecaf 编译器架构分析

Intro

对于一个 MiniDecaf 程序，本实验实现的编译器流程基本如上图所示；其中每一个阶段所实现的具体功能分布在本学期课程讲授内容之中，以下是简要概括：

  • 阶段一（词法语法分析器）：自动构造工具实现词法和语法分析程序。借助 Lex/Yacc 实现词法和语法分析一遍扫描源程序后直接产生一种高级中间表示（实验指定的抽象语法树
    AST）。关于 Lex/ply 的使用细节可以查看文档，

    • 实验框架中已经完整实现词法语法分析的核心部分，能够实现扫描源程序，并且直接生成抽象语法树的基础版本（仅仅支持带 return 的 main 函数）
    • 词法语法分析的过程鼓励查看文档进行理解

  • 实验框架中阶段一的基础已经搭建好，需要我们完善的是补充相关“语法规则”，“符号”等以支持相应 minidecaf 程序语句的语法分析。例如 for、dowhile 循环，原始编译器是无法识别的，我们需要做的是在基础上不断完善该编译器的功能。因此，在这一阶段，应着重理解如何添加新的语法规则？如何引入新的标识符以支持扫描？

  • 阶段二：语义分析。遍历抽象语法树构造符号表、实现静态语义检查（非上下文无关语法检查以及类型检查等），产生带标注的抽象语法树。这一阶段将对抽象语法树 AST 进行两遍扫描：第一遍扫描完成符号表的建立，并且检测符号声明冲突以及跟声明有关的符号引用问题（例如，函数 A 调用函数 B，但是函数 B 没有定义）；第二遍扫描检查所有的语句以及表达式的参数的数据类型。

    • 本阶段需要着重理解语义分析所完成的任务：构造符号表+静态语义检查。符号表的构建是为了存储程序中遇到的变量、函数、标识符等，是扫描程序构造基本语义的第一步。第二步强调语义检查，对于程序中出现的不符合标准的变量调用、声明、类型错误进行检查，由于本实验实现的 minidecaf 编译器只要求支持 int 类型的变量，因此可以略过类型检查这一部分。

  • 阶段三：中间代码生成。将带标注的抽象语法树（decorated AST）所表示的输入程序翻
    译成适合后期处理的另一种中间表示方式，即三地址码 TAC，并在合适的地方加入诸如检
    查数组访问越界、数组大小非法等运行时错误处理的内容。阶段三完成后所得的三地址码程
    序可在 TAC 模拟器上执行。

  • 阶段四：中间代码优化。主要是基于 TAC 代码实现简单的数据流分析（如活跃变量数
    据流、到达-定值数据流、UD 链和 DU 链等）模块，也可以进行简单的优化（如常量合并、
    常量传播、公共子表达式消除等）。经过数据流分析所求得的活跃变量信息，后续的寄存器
    分配具有重要支撑作用。

  • 阶段五：目标代码生成。实验框架包括汇编指令选择、寄存器分配和栈帧管理。实验框
    架中包含一个面向 RISC-V 架构的后端，可生成适合实际 RISC-V 机器上的汇编代码。这些
    RISC-V 汇编代码也可以利用开源模拟处理器 QEMU [6] 或 Spike [7] 进行模拟执行。这一
    阶段的实验框架仅实现了一种启发式的寄存器分配算法。如果同学们有兴趣，可以思考如何
    实现更加高效的寄存器分配算法（比如，基于图着色方法的寄存器分配算法），并生成正确
    的汇编代码

parser 类所完成的功能

ply_parser 类具体完成对于整个 token 流的顺序扫描，并且有效的转化为 AST 树，利用 AST 类中定义好的节点类、函数等等，将每一个 token 所传递的源代码的结构抽象成树。这也是语法分析的关键部分。

首先我们看 plyparser 类，里面每一条以”p”开头的函数都定义了一条针对 token 流的语法规则，或者说产生式，可以将 ply_parser 想象成一台有限状态机，不断通过文法来匹配 token。

以 p_return 为例，这里对应到 parser 扫描到 minidecaf 程序当中 return 语句时，应该执行的 AST 转换操作。dotstring 中定义的是文法（具体可见形式语言与状态机）它规定了我们遇到特定的 token 应该匹配到的产生式（终结符），见到此处熟悉的”Return“，正是我们之前在词法分析（lex）中定义的关键字，此处发挥作用。

对于匹配到的文法，函数的具体内容定义了需要执行的操作，例如此处将 return 语句中对应的 statement 表达式，构造一个新的 Return 类并且加入 AST 树当中，经过前面若干步类似的操作，整棵 AST 树得以构建。

💡 我们需要做什么？ 在此处为支持新添加的代码功能例如for循环，我们需要定义新的AST节点，并且思考它应该继承或者运用到哪些AST节点类。并且需要告诉parser如何识别到for循环语句，定义新的文法，并且将相应的token纳入for语句的”cond“、”stmt“等等类，构造出一个新的AST节点加入树中。

至此，前端的工作基本完成。

Python 代码中端代码分析

中端完成的任务主要包含语义分析和中间代码（TAC）生成部分。

语义分析

首先来看语义分析，书接上回，在构造好整个源程序的 AST 树之后，接下来需要完成的工作就是对整棵树进行扫描从而进行语义分析。在规范的流程中，需要对抽象语法树进行两次扫描，分别进行符号表的构建和代码的类型检查，因为 minidecaf 编译器仅要求支持 int 类型的变量，因此我们只需要关心符号表构建的过程。

同样的，在对 AST 树扫描的过程中，我们依然使用 visitor 设计模式，在./frontend/typecheck 文件夹下的 namer.py 主要实现扫描，其中对于需要扫描到的节点，我们可以重载它的 visitor 类对应的成员函数，然后在其中填上我们在这次扫描中需要对该节点进行的操作：

transform 函数是整个扫描的入口，它的这个入口就是整个抽象语法树的根节点，也就是 program 类。传进来后接下来调用 program 的 accept 函数，然后依次调用并且访问整个子树。

其次，在 transform 函数中出现了 global scope 类，这个是全局作用域所对应的符号表，它存储的是全局变量以及各种函数的符号，目前为止我们有且只有一个 main 函数作用域，因此实现较简单，后面引入函数、全局变量后，就需要考虑全局作用域的问题。
